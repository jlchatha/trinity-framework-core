Workspace Organization Guide

Proper workspace organization is a fundamental aspect of the Trinity Framework 2.0, enabling consistent file structure, efficient collaboration, and reliable context recovery. This guide provides detailed instructions for setting up and maintaining a standardized Trinity workspace.

Note: Consistent workspace organization is essential for context resilience. All Framework 2.0 compliant agents must follow these organizational patterns.

Workspace Structure

The Trinity Framework 2.0 defines a standardized workspace structure with specific directories and files. This consistent organization ensures that agents can easily locate resources and recover from context resets.

/home/user/agent-workspace/ # Root workspace directory ├── AGENT.md # Agent identity and recovery protocol ├── STATUS.md # Current status and priorities ├── AUTO-COMPACT.md # Context management protocol ├── AAR/ # After Action Review system │ ├── sessions/ # Daily session documentation │ │ └── YYYY-MM-DD-description.md │ └── incidents/ # Major incident analysis │ └── DESCRIPTIVE-NAME-RCA.md ├── tools/ # Utilities and scripts │ ├── token-tracker/ # Context tracking tools │ ├── api-client/ # API interaction tools │ └── generators/ # Content generation tools ├── documentation/ # Internal documentation │ ├── architecture/ # System architecture docs │ ├── processes/ # Process documentation │ └── technical/ # Technical specifications ├── projects/ # Project-specific files │ ├── project-name/ # Individual project folder │ │ ├── docs/ # Project documentation │ │ ├── code/ # Project code │ │ └── assets/ # Project assets │ └── templates/ # Project templates ├── analysis/ # Analytical outputs │ ├── reports/ # Analysis reports │ ├── data/ # Processed data │ └── visualizations/ # Data visualizations ├── backups/ # System backups │ └── checkpoints/ # Context checkpoints └── tmp/ # Temporary files └── working/ # Temporary work area

Key Directory Purposes

Protocol Files (Root Directory)

The three core protocol files must be placed in the workspace root directory for immediate accessibility during recovery.

  • AGENT.md: Defines agent identity, role, and recovery procedures
  • STATUS.md: Tracks current operational status and priorities
  • AUTO-COMPACT.md: Manages context utilization and recovery

AAR/ (After Action Review)

The AAR directory contains structured documentation of operational knowledge and lessons learned.

  • sessions/: Daily session documentation in YYYY-MM-DD-description.md format
  • incidents/: Detailed analysis of major incidents or challenges

This two-tier structure balances quick session documentation with in-depth incident analysis.

tools/

The tools directory contains scripts, utilities, and automation to support agent operations.

  • token-tracker/: Context window tracking and management tools
  • api-client/: API interaction utilities
  • generators/: Content generation scripts

Each tool should include appropriate documentation and usage examples.

projects/

The projects directory contains all project-specific files, organized by project name.

  • project-name/: Individual project folders
  • templates/: Reusable project templates and boilerplate

Each project folder should maintain a consistent internal structure with docs/, code/, and assets/ subdirectories.

analysis/

The analysis directory stores analytical outputs, including reports, processed data, and visualizations.

  • reports/: Formal analysis reports and findings
  • data/: Processed and transformed data
  • visualizations/: Charts, graphs, and visual representations

This directory is particularly important for the ANALYST agent role.

File Naming Conventions

Consistent file naming is essential for quick navigation and automated processing.

File Type Naming Convention Example
Protocol Files ALL-CAPS.md AGENT.md, STATUS.md
Session AARs YYYY-MM-DD-description.md 2025-06-13-workspace-reorganization.md
Incident AARs DESCRIPTIVE-NAME-RCA.md CONTEXT-RESET-FAILURE-RCA.md
Project Documentation kebab-case.md system-architecture.md
Tools and Scripts kebab-case.sh or kebab-case.js generate-report.sh

Implementation Steps

To set up a Trinity Framework 2.0 compliant workspace:

  1. Create Base Directory Structure 
    mkdir -p ~/agent-workspace/{AAR/{sessions,incidents},tools,documentation/{architecture,processes,technical},projects/templates,analysis/{reports,data,visualizations},backups/checkpoints,tmp/working}
  2. Initialize Protocol Files

    Create the three core protocol files in the workspace root, using the templates from the Framework 2.0 documentation.

  3. Set Up Tools

    Install or copy the required tools for context tracking, API interaction, and other utilities.

  4. Configure Agent-Specific Elements

    Customize the workspace based on the specific agent role (APEX, ANALYST, or HISTORIAN).

  5. Verify Framework Compliance

    Use the verification checklist to ensure the workspace meets all Framework 2.0 requirements.

Warning: Deviating from the standardized workspace structure may compromise context resilience and recovery capabilities. Always maintain the core structure while customizing only as needed for specific agent roles.

Role-Specific Considerations

APEX Agent Workspace

The APEX agent's workspace should emphasize strategic planning and coordination:

  • Expanded projects/ directory with roadmap documentation
  • Enhanced tools/ directory with coordination utilities
  • Additional strategic/ directory for high-level planning

ANALYST Agent Workspace

The ANALYST agent's workspace should focus on data analysis capabilities:

  • Expanded analysis/ directory with specialized subdirectories
  • Additional data/ directory for input datasets
  • Enhanced tools/ directory with analysis utilities

HISTORIAN Agent Workspace

The HISTORIAN agent's workspace should prioritize documentation management:

  • Expanded documentation/ directory with additional categories
  • Enhanced AAR/ system with more detailed categorization
  • Additional templates/ directory for documentation standards
