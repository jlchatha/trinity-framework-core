Glossary of Terms

This glossary provides definitions for key terms and concepts used throughout the Trinity Framework 2.0 Core documentation.

A C F I M O P R S T U W
A
AAR (After Action Review) System

A two-tier documentation system for preserving operational knowledge across context resets. Consists of session AARs (brief daily documentation) and incident AARs (detailed analysis of major challenges).

See: AAR System Guide

AGENT.md Protocol File

One of the three core protocol files that defines an agent's identity, role, responsibilities, and recovery procedures. This is the primary file referenced during auto-compact recovery.

See: Protocol Files Reference

ANALYST Agent Role

One of the three specialized agent roles in the Trinity Framework, focused on data analysis, technical problem-solving, and providing evidence-based recommendations.

See: Agent Roles Reference

APEX Agent Role

The strategic coordination agent role in the Trinity Framework, responsible for project leadership, resource allocation, and high-level decision-making.

See: Agent Roles Reference

Auto-Compact Operation

The process of clearing the context window when it approaches capacity limits, requiring state restoration through the recovery protocol.

See: Context Management Guide

AUTO-COMPACT.md Protocol File

One of the three core protocol files that manages context utilization and provides recovery mechanisms. Contains context metrics, checkpoint data, and recovery instructions.

See: Protocol Files Reference

C
Checkpoint Context Management

A serialized state snapshot created at specific threshold points (NOTICE, WARNING, ALERT) to enable recovery after context resets. Contains utilization data, timestamp, and operation history.

See: Token Tracking Guide

Claude Technology

The large language model developed by Anthropic that powers the Trinity Framework. The framework is specifically designed for Claude's capabilities and context management requirements.

Context Resilience Principle

A core principle of the Trinity Framework focused on maintaining operational continuity across context resets through standardized recovery procedures.

See: Context Resilience Principle

Context Window LLM Concept

The limited amount of information that Claude can "remember" during a conversation, measured in tokens. The Trinity Framework's token tracking system helps manage this limitation.

See: Token Tracking Guide

HISTORIAN Agent Role

One of the three specialized agent roles in the Trinity Framework, focused on documentation management, knowledge preservation, and maintaining historical records.

See: Agent Roles Reference

I
Incident AAR Documentation

A comprehensive analysis document (100+ lines) created after major incidents or challenges, providing detailed technical documentation for knowledge transfer.

See: AAR System Guide

M
MILESTONE Project Term

A project term representing an achievement checkpoint, typically spanning 7 UNITs of work (approximately equivalent to a week in traditional terminology).

Multi-Agent Collaboration Principle

A core principle of the Trinity Framework enabling specialized agents (APEX, ANALYST, HISTORIAN) to work together effectively as a cohesive system through standardized interactions.

See: Multi-Agent Collaboration Principle

O
Operation Types Context Management

Categorized actions that consume context window space, such as USER_MEDIUM, ASSISTANT_LARGE, TOOL_READ_SMALL, etc. Used by the token tracking system to estimate token usage.

See: Token Tracking Guide

P
PHASE Project Term

A project term representing a major project segment, typically spanning 4 MILESTONEs (approximately equivalent to a month in traditional terminology).

PROJECT Project Term

A project term representing a complete initiative, typically spanning 3 PHASEs (approximately equivalent to a quarter in traditional terminology).

Protocol Files System

The three core files (AGENT.md, STATUS.md, AUTO-COMPACT.md) that form the foundation of the Trinity Framework, providing standardized structure for agent operation and recovery.

See: Protocol Files Reference

Protocol-Based Recovery Principle

A core principle of the Trinity Framework focused on using standardized procedures for restoring operational state after context resets.

R
Recovery Protocol Procedure

The standardized procedure for restoring operational awareness after a context reset, including reading protocol files, checking system status, and resuming work.

See: Context Resilience Principle

S
Session AAR Documentation

A brief documentation (20-50 lines) created after daily work sessions, covering what worked, what didn't, lessons learned, and next steps. Named with the pattern YYYY-MM-DD-description.md.

See: AAR System Guide

STATUS.md Protocol File

One of the three core protocol files that tracks current operational status, priorities, and progress. Updated after every work session to maintain temporal context.

See: Protocol Files Reference

Systematic Documentation Principle

A core principle of the Trinity Framework focused on maintaining documentation in standardized formats and locations for consistency and accessibility.

â†‘ Back to top
T
Thresholds Context Management

Defined context utilization levels that trigger specific actions:

  • NOTICE (65%): Creates a checkpoint for potential future recovery
  • WARNING (75%): Recommends completing current task before starting new complex operations
  • ALERT (85%): Suggests performing auto-compact to prevent context truncation

See: Token Tracking Guide

Token LLM Concept

The basic unit of text processing in language models like Claude. Approximately 3-4 characters or 0.75 words per token. The context window capacity is measured in tokens.

See: Token Tracking Guide

Token Tracking System

The system for monitoring context window utilization, creating checkpoints, and enabling recovery after context resets. A core capability of the Trinity Framework 2.0.

See: Token Tracking Guide

Trinity Framework System

A comprehensive framework for building context-resilient agent systems with Claude, featuring standardized protocol files, specialized agent roles, and context management capabilities.

Trinity Model Architecture

The three-agent architecture (APEX, ANALYST, HISTORIAN) that forms the foundation of the Trinity Framework, enabling specialized roles to work together effectively.

See: Multi-Agent Collaboration Principle
