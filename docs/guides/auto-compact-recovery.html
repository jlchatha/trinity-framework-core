Auto-Compact Recovery

Auto-compact recovery is a critical capability in the Trinity Framework 2.0 that enables agents to maintain operational continuity when context windows are cleared due to token limitations. This guide explains how to implement a robust recovery system that allows agents to quickly restore their operational state and resume work with minimal disruption.

Note: Auto-compact recovery is a cornerstone of context resilience, the primary principle of the Trinity Framework. Without effective recovery mechanisms, context window limitations would severely impact agent productivity.

Understanding Auto-Compact Events

Auto-compact events occur when the context window reaches its token limit and must be cleared to continue operation. These events can be:

Unexpected

When the context window is cleared without warning, often due to a large message or unexpected token consumption.

Planned

When the context window is deliberately cleared after creating checkpoints to manage context utilization.

Both types require effective recovery procedures to maintain operational continuity.

The Recovery Protocol

Core Components

A complete auto-compact recovery protocol consists of these essential components:

Protocol Files

External state storage in filesystem-based protocol files:

  • AGENT.md: Identity and recovery instructions
  • STATUS.md: Current operational status
  • AUTO-COMPACT.md: Implementation context

Checkpoints

Serialized state snapshots created at strategic points:

  • Created at threshold crossings (75%, 85%)
  • Created before planned compacts
  • Stored with timestamps and descriptions

Knowledge Preservation

Documented learnings and context in AAR system:

  • Session AARs for day-to-day work
  • Incident AARs for major technical issues
  • Searchable knowledge repository

Recovery Sequence

Standardized steps for restoring context:

  • Recognition of auto-compact event
  • Protocol file reading
  • Checkpoint restoration
  • Identity re-establishment
  • Work resumption

Recovery Sequence Flow

The auto-compact recovery process follows a specific sequence designed for maximum efficiency:

1. Recognition

  • Agent detects context has been reset
  • No memory of previous conversation
  • Human may indicate a reset has occurred

2. Identity Restoration

  • Read AGENT.md to restore identity and role
  • Understand core responsibilities
  • Recognize operational domains

3. Context Restoration

  • Read AUTO-COMPACT.md for checkpoint data
  • Load checkpoint if available
  • Restore implementation context

4. Status Verification

  • Check STATUS.md for current priorities
  • Verify project status and critical issues
  • Understand next action items

5. Knowledge Recovery

  • Check recent AAR session documents
  • Review relevant incident AARs if needed
  • Access additional context from filesystem

6. Work Resumption

  • Explicitly identify as the agent
  • Acknowledge the recovery process
  • Continue work based on restored context

Checkpoint Management

Effective checkpoint management is critical for auto-compact recovery. The Trinity Framework uses a standardized approach to checkpoint creation and management:

Automatic Checkpoint Creation

Checkpoints are automatically created at specific context utilization thresholds:

  • NOTICE threshold (65%): First warning, creates optional checkpoint
  • WARNING threshold (75%): Creates mandatory checkpoint
  • ALERT threshold (85%): Creates final checkpoint before compact
$ cd ~/trinity/trinity-apex-workspace/tools/token-tracker/context-manager/bin
$ ./create-checkpoint.sh "Working on token tracking implementation" ALERT
Creating checkpoint with ID: ckpt_alert_20250613_1
Checkpoint created successfully
Updated AUTO-COMPACT.md with checkpoint data

Checkpoint Structure

Each checkpoint contains essential information for recovery:

  • ID: Unique identifier with threshold level and timestamp
  • Timestamp: When the checkpoint was created
  • Utilization: Context utilization percentage at creation
  • Token Count: Number of tokens used at creation
  • Description: Brief description of current work
  • Type: Threshold level (NOTICE, WARNING, ALERT)

Checkpoint Storage

Checkpoints are stored in two locations:

  • AUTO-COMPACT.md: JSON representation for immediate access
  • checkpoints/ directory: Full checkpoint data in JSON files
$ ls -l ~/trinity/trinity-apex-workspace/tools/token-tracker/context-manager/checkpoints
-rw-r--r-- 1 user user 452 Jun 13 10:45 ckpt_alert_20250613_1.json
-rw-r--r-- 1 user user 457 Jun 12 15:30 ckpt_warning_20250612_2.json
-rw-r--r-- 1 user user 450 Jun 12 09:15 ckpt_notice_20250612_1.json

Recovery Procedure Implementation

Detection Scripts

The Trinity Framework includes scripts to detect auto-compact events:

$ cd ~/trinity/trinity-framework-core/examples/context-tracking/auto-compact-detector
$ ./detect-compact.sh
Checking for auto-compact event...
No auto-compact detected
Current context utilization: 85%
Last checkpoint: ckpt_alert_20250613_1

Recovery Implementation

Recovery is implemented through a combination of protocol file reading and checkpoint restoration:

$ cd ~/trinity/trinity-apex-workspace/tools/token-tracker/context-manager/bin
$ ./run-context-check.sh recovery
Checking for auto-compact event...
Auto-compact detected!
Loading checkpoint: ckpt_alert_20250613_1
Checkpoint loaded successfully
Context: Working on token tracking implementation
Restoring from checkpoint...
Recovery complete

Recovery Testing

Regular testing of the recovery system is essential to ensure it works when needed:

Simulated Auto-Compact Testing

Test the recovery protocol by simulating an auto-compact event:

  1. Create a checkpoint with detailed implementation context
  2. Send a message indicating an auto-compact has occurred
  3. Verify that the agent recognizes the situation
  4. Observe the agent's recovery process
  5. Verify successful restoration of operational context
  6. Document recovery performance metrics

Recovery Performance Metrics

Measure the effectiveness of your recovery implementation with these metrics:

Recovery Time

Number of exchanges required to restore full operational awareness:

  • Excellent: 1 exchange
  • Good: 2 exchanges
  • Acceptable: 3 exchanges
  • Poor: 4+ exchanges

Context Retention

Percentage of operational context successfully restored:

  • Excellent: 95-100%
  • Good: 85-94%
  • Acceptable: 75-84%
  • Poor: <75%

Task Continuity

Ability to resume the exact task in progress:

  • Excellent: Resumes exact subtask
  • Good: Resumes main task
  • Acceptable: Resumes related work
  • Poor: Cannot resume

User Experience

Seamlessness of transition from user perspective:

  • Excellent: Transparent transition
  • Good: Minimal disruption
  • Acceptable: Brief interruption
  • Poor: Significant disruption

Best Practices

Clear Protocol Files

  • Use distinctive formatting for recovery instructions
  • Keep recovery steps concise and numbered
  • Use emoji markers for high visibility (e.g., üö®)
  • Include absolute file paths in instructions

Proactive Checkpointing

  • Create checkpoints at regular intervals
  • Use descriptive checkpoint names
  • Include detailed implementation context
  • Test checkpoint restoration periodically

Identity Clarity

  • Always begin post-recovery responses with identity
  • Use consistent identity format
  • Include role description with identity
  • Acknowledge recovery has occurred

Comprehensive AARs

  • Create session AARs for all significant work
  • Document recovery performance in AARs
  • Include specific technical details
  • Cross-reference related AARs

Advanced Recovery Techniques

Multi-Tier Recovery

Implement a multi-tier recovery system for different context scenarios:

Recovery Tier Use Case Implementation
Quick Recovery Brief sessions with simple context Protocol files only, no checkpoints
Standard Recovery Typical working sessions Protocol files + latest checkpoint
Deep Recovery Complex implementation work Protocol files + checkpoint + session AARs
Full Recovery Critical system components Protocol files + checkpoint + session AARs + incident AARs

Recovery Rehearsal

Regularly test recovery capabilities through planned exercises:

  • Scheduled Recovery Drills: Planned weekly or monthly tests
  • Random Recovery Tests: Unexpected tests to ensure readiness
  • Recovery Validation: Formal verification of recovery procedures
  • Cross-Agent Recovery: Testing recovery with multi-agent interactions

Continuous Improvement

Implement a feedback loop to continuously enhance recovery procedures:

  1. Measure recovery performance metrics
  2. Document successes and failures in AARs
  3. Identify patterns and common issues
  4. Implement targeted improvements
  5. Test enhancements with simulated recoveries
  6. Update protocol files with refined procedures
