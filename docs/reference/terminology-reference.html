Terminology Reference

This comprehensive reference guide defines all key terminology used in the Trinity Framework 2.0 Core. Understanding these terms is essential for effective implementation and use of the framework. Terms are organized by category for easier reference.

Framework Architecture Agent System Context Management Knowledge Preservation Operational Terminology Project Management

Framework Architecture

Trinity Framework

A comprehensive architecture for building context-resilient agent systems using LLMs. The framework provides structured protocols, standardized file organization, and recovery mechanisms to maintain operational continuity across context limitations.

Related Terms: Framework 2.0, Context Resilience, Protocol Files

Framework 2.0

The second major version of the Trinity Framework, introducing standardized context management, unified agent protocols, and the Trinity agent model. Framework 2.0 represents a complete standardization of workspace structure and operational procedures.

Related Terms: Trinity Framework, Trinity Model, Workspace Standardization

Trinity Model

The standard three-agent architecture used in the Trinity Framework, comprising APEX, ANALYST, and HISTORIAN agents working collaboratively with clearly defined roles and responsibilities. This model provides specialized expertise while maintaining coordinated operation.

Related Terms: APEX Agent, ANALYST Agent, HISTORIAN Agent, Multi-Agent Collaboration

Core Principles

The fundamental design principles of the Trinity Framework 2.0, including Context Resilience, Workspace Standardization, Protocol-Based Recovery, Systematic Documentation, and Operational Autonomy. These principles guide all framework implementation decisions.

Related Terms: Context Resilience, Workspace Standardization, Protocol-Based Recovery, Systematic Documentation, Operational Autonomy

Protocol Files

The core standardized files (AGENT.md, STATUS.md, AUTO-COMPACT.md) that define agent identity, track operational status, and manage context utilization. These files are essential for maintaining operational continuity across context resets.

Related Terms: AGENT.md, STATUS.md, AUTO-COMPACT.md, Protocol-Based Recovery

Agent System

APEX Agent

The strategic coordinator in the Trinity Model, responsible for system architecture, integration patterns, and cross-agent coordination. APEX has the highest level of decision-making authority and provides direction to other agents.

Related Terms: Trinity Model, Operational Autonomy, Strategic Planning

ANALYST Agent

The analytical specialist in the Trinity Model, responsible for data analysis, performance optimization, and implementation work. ANALYST provides insights based on system data and implements specific components.

Related Terms: Trinity Model, Operational Autonomy, Performance Analysis

HISTORIAN Agent

The documentation specialist in the Trinity Model, responsible for maintaining comprehensive documentation, managing the knowledge base, and preserving historical context. HISTORIAN ensures all system knowledge is accessible and current.

Related Terms: Trinity Model, Operational Autonomy, Systematic Documentation

Multi-Agent Collaboration

The coordinated operation of multiple specialized agents within the Trinity Framework, each with defined roles and domains of expertise. This collaboration pattern enables complex work to be distributed across specialized agents while maintaining overall system coherence.

Related Terms: Trinity Model, Operational Autonomy, Agent Coordination

Agent Coordination

The mechanisms and protocols for maintaining alignment and coherent operation across multiple agents. This includes coordination sections in STATUS.md, handoff procedures, and conflict resolution protocols.

Related Terms: Multi-Agent Collaboration, STATUS.md, Handoff Procedures

Context Management

Context Resilience

The ability to maintain operational continuity across context resets or limitations. This is the primary principle of the Trinity Framework and is achieved through standardized protocols, checkpointing, and knowledge preservation mechanisms.

Related Terms: Auto-Compact, Checkpointing, Protocol-Based Recovery

Auto-Compact

The process where a large language model's context window is cleared to manage token limitations. This can be either an automatic process triggered by the system or a planned operation to manage context utilization.

Related Terms: Context Resilience, Auto-Compact Recovery, Token Tracking

Auto-Compact Recovery

The standardized process for restoring operational context after a context window reset. This includes reading protocol files, loading checkpoints, and re-establishing agent identity and operational awareness.

Related Terms: Auto-Compact, Protocol-Based Recovery, Checkpointing

Token Tracking

The system for monitoring and managing context utilization through token counting, thresholds, and visualization. This provides early warning of impending context limitations and triggers checkpoint creation at specific thresholds.

Related Terms: Context Utilization, Thresholds, Checkpointing

Checkpointing

The process of creating serialized state snapshots at strategic points during operation, especially at context utilization thresholds. Checkpoints contain essential information for recovery after context resets.

Related Terms: Thresholds, Auto-Compact Recovery, Checkpoint Restoration

Context Utilization

The percentage of the maximum context window capacity currently in use. This is tracked through token counting and visualized to monitor approaching limitations.

Related Terms: Token Tracking, Thresholds, Context Visualization

Thresholds

Specific context utilization percentages that trigger actions such as checkpointing or recovery preparation. Standard thresholds are NOTICE (65%), WARNING (75%), and ALERT (85%).

Related Terms: Context Utilization, Checkpointing, NOTICE Threshold, WARNING Threshold, ALERT Threshold

NOTICE Threshold

The first context utilization threshold (typically 65%) that provides an early indication of context accumulation and triggers optional checkpointing. This threshold serves as an awareness signal without requiring immediate action.

Related Terms: Thresholds, Context Utilization, Checkpointing

WARNING Threshold

The second context utilization threshold (typically 75%) that indicates moderate context accumulation and triggers mandatory checkpointing. At this level, agents should complete critical work and prepare for possible context management.

Related Terms: Thresholds, Context Utilization, Checkpointing

ALERT Threshold

The third and most critical context utilization threshold (typically 85%) that indicates high context accumulation and triggers final checkpoint creation. At this level, agents should prepare for imminent context reset.

Related Terms: Thresholds, Context Utilization, Checkpointing, Auto-Compact

Knowledge Preservation

AAR System

After Action Review system, a structured knowledge preservation mechanism used to document experiences, lessons learned, and technical details across context boundaries. The system uses a two-tier approach with session and incident AARs.

Related Terms: Session AAR, Incident AAR, Knowledge Preservation

Session AAR

A lightweight documentation file created after significant work sessions to preserve key learnings and context across sessions. These AARs follow the format YYYY-MM-DD-description.md and focus on what worked, what didn't, and lessons learned.

Related Terms: AAR System, Knowledge Preservation, Context Resilience

Incident AAR

A comprehensive documentation file created after major incidents or technical challenges to preserve detailed knowledge. These AARs follow the format DESCRIPTIVE-NAME-RCA.md and include root cause analysis and detailed technical information.

Related Terms: AAR System, Knowledge Preservation, Root Cause Analysis

Systematic Documentation

The principle of creating and maintaining documentation using standardized formats, locations, and processes. This ensures knowledge is accessible, consistent, and preserved across context boundaries.

Related Terms: AAR System, Documentation Standards, HISTORIAN Agent

Knowledge Preservation

The practice of systematically documenting and storing information to maintain operational knowledge across context boundaries. This is achieved through protocol files, AARs, and structured documentation.

Related Terms: AAR System, Context Resilience, Systematic Documentation

Operational Terminology

AGENT.md

The primary protocol file that defines agent identity, role, responsibilities, and recovery procedures. Located in the root of each agent workspace, this file is the first reference during recovery.

Related Terms: Protocol Files, Auto-Compact Recovery, Agent Identity

STATUS.md

The protocol file that tracks current operational status, project progress, and priorities. Located in the root of each agent workspace, this file provides ongoing awareness of current work and next steps.

Related Terms: Protocol Files, Auto-Compact Recovery, Priorities

AUTO-COMPACT.md

The protocol file that manages context tracking, checkpoint data, and implementation context. Located in the root of each agent workspace, this file is essential for context management and recovery.

Related Terms: Protocol Files, Auto-Compact Recovery, Checkpointing

Workspace Standardization

The principle of maintaining consistent file structure, directory organization, and naming conventions across all agent workspaces. This standardization ensures predictable file locations and organization.

Related Terms: Protocol Files, Directory Structure, Naming Conventions

Protocol-Based Recovery

The principle of using standardized, documented procedures to recover from context resets. This approach ensures consistent, reliable recovery through clearly defined steps and external state storage.

Related Terms: Auto-Compact Recovery, Protocol Files, Checkpointing

Operational Autonomy

The principle of enabling agents to operate independently within their domain expertise while maintaining coordination with other agents. This balances specialist expertise with system-wide coherence.

Related Terms: Multi-Agent Collaboration, Agent Coordination, Trinity Model

Project Management

UNIT

The atomic operational cycle in the Trinity Framework project management terminology, equivalent to a daily work cycle. This is the smallest formal temporal measurement unit in the framework.

Related Terms: MILESTONE, PHASE, PROJECT

MILESTONE

An achievement checkpoint in the Trinity Framework project management terminology, equivalent to a weekly work cycle (7 UNITs). Milestones represent significant progress markers within a PHASE.

Related Terms: UNIT, PHASE, PROJECT

PHASE

A major project segment in the Trinity Framework project management terminology, equivalent to a monthly work cycle (4 MILESTONEs). PHASEs represent significant stages in a PROJECT's lifecycle.

Related Terms: UNIT, MILESTONE, PROJECT

PROJECT

A complete initiative in the Trinity Framework project management terminology, equivalent to a quarterly work cycle (3 PHASEs). PROJECTs represent complete, self-contained bodies of work.

Related Terms: UNIT, MILESTONE, PHASE

ITERATION

A process improvement cycle in the Trinity Framework project management terminology, equivalent to a fortnightly work cycle (14 UNITs). Iterations focus on incremental improvements within a MILESTONE.

Related Terms: UNIT, MILESTONE, PHASE
